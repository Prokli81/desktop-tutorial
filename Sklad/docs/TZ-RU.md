# Техническое задание: склад мебели и бытовой техники

**Проект:** Sklad (рабочее название)  
**Версия документа:** 1.2  
**Дата:** 10.06.2026  
**Статус:** согласовано с заказчиком (готово к разработке MVP)

---

## 1. Цель проекта

Создать единую систему учёта склада для **мебели** (в разборе по пакетам-комплектующим) и **бытовой техники**, в которой:

- **телефон** используется для сканирования этикеток на складе;
- **компьютер** используется для подтверждения операций, накладных и отчётов;
- **облако** обеспечивает синхронизацию между устройствами;
- при недоступности облака возможна работа через **локальную сеть** (телефон и ПК в одной Wi‑Fi / hotspot).

Система должна ощущаться как **одно целое**, а не как две отдельные программы.

---

## 2. Предметная область

### 2.1. Мебель

- Мебель (комод, тумба, шкаф и т.д.) поставляется и хранится в виде **отдельных пакетов** (коробок с комплектующими).
- **Один тип пакета** может подходить к **разным изделиям** (например, один пакет может идти и к комоду, и к тумбе, и к шкафу).
- Для сборки изделия нужен **набор пакетов** по «рецепту комплекта» (сколько каких пакетов, какого цвета).
- На каждой упаковке — **своя уникальная этикетка**.

### 2.2. Бытовая техника

- Техника учитывается по той же общей логике: **скан → уточнение → привязка к накладной**.
- Чаще всего одна коробка = один товар, но схема сканирования и подтверждения **та же**, что и для мебели.

### 2.3. Маркировка

На каждой упаковке — **уникальная** этикетка. Формат может быть любым из перечисленных:

| Тип | Пример |
|-----|--------|
| Штрихкод | EAN-13, Code128 |
| QR-код | Ссылка или текст внутри |
| Текст (буквы и цифры) | `KOM-204-DUB` (модель + цвет) |

Буквенно-цифровой код несёт не только модель, но и **цвет изделия** (или пакета).

**Согласованный формат (базовый):** `КОД-МОДЕЛЬ-ЦВЕТ`, например `KOM-204-DUB` (комод, модель 204, цвет дуб).

**Редактирование:** правила разбора кода и справочники сокращений (что значит `KOM`, `DUB` и т.д.) должны **настраиваться и редактироваться** на ПК. Если формат этикеток изменится — правила обновляются без переписывания программы.

## 3. Участники и роли

| Роль | Устройство | Основные действия |
|------|------------|-------------------|
| Кладовщик / оператор на складе | Телефон (**Android**) | Сканирование, создание предварительных запросов, просмотр модели и остатка |
| Ответственный за учёт | ПК | Подтверждение и отклонение запросов, накладные, справочники, отчёты |

> **Важно:** окончательное подтверждение приёмки и отгрузки выполняется **только на ПК**. Телефон не изменяет остатки напрямую.

## 4. Основные сценарии

### 4.1. Поступление товара (приёмка)

```
Телефон:  скан этикетки → запрос попадает в накладную (статус: ожидает)
ПК:       оператор проверяет всю накладную → «Подтвердить всю накладную» / Отклонить
          после подтверждения → все позиции накладной принимаются на склад
Телефон:  уведомление «Накладная подтверждена» или «Отклонена»
```

Дополнительно:

- сразу после скана или позже можно **уточнить**, что это за товар (модель, цвет, привязка к накладной);
- допускается режим: **сначала отсканировать все коробки**, потом разобрать по накладной;
- **подтверждение целиком по накладной** — основной режим; при необходимости позже можно **доделать** подтверждение **по одной позиции** (резерв на будущие версии).

### 4.2. Отгрузка товара

```
Телефон:  скан этикетки → показать модель и текущий остаток
          → позиция добавляется в накладную отгрузки (статус: ожидает)
ПК:       оператор указывает номер заказа и клиента
          → проверяет накладную → «Подтвердить всю накладную» / Отклонить
          после подтверждения → остатки уменьшаются по всем позициям
Телефон:  уведомление о результате
```

**Обязательные поля при отгрузке на ПК:** номер заказа и клиент.

### 4.3. Просмотр товара по скану

При наведении камеры на этикетку (без операции приёмки/отгрузки) система показывает:

- найден ли товар в базе;
- модель, цвет (если известны);
- текущий статус (на складе, в ожидании, отгружен и т.д.);
- остаток по данной модели (где применимо).

### 4.4. Мебель: комплектация по накладной

**Сценарий «приняли пакеты → разложили по изделию»:**

1. При поступлении кладовщик сканирует **все пакеты** поставки.
2. На ПК (или после уточнения) вводится позиция накладной: например, «Комод модель 204, цвет дуб, 1 шт.».
3. Система **автоматически подбирает** из принятых пакетов те, что подходят к этому изделию.
4. Система показывает:
   - какие пакеты уже привязаны к изделию;
   - **каких пакетов не хватает** для полного комплекта;
   - какие пакеты остаются нераспределёнными.

**Обратный сценарий «нужно собрать изделие»:**

1. Оператор выбирает изделие: «Шкаф модель X, цвет Y».
2. Система показывает список необходимых пакетов: **количество, тип, цвет**.
3. Кладовщик использует список для поиска пакетов на складе.

### 4.5. Бытовая техника: скан и уточнение

1. Скан этикетки при поступлении → предзапрос на приёмку.
2. Уточнение: «Холодильник Samsung X» (вручную или из справочника / по накладной).
3. При отгрузке — скан → показ модели и остатка → предзапрос → подтверждение накладной на ПК (с заказом и клиентом).

### 4.6. Подтверждение накладной на ПК (согласовано)

| Правило | Решение |
|---------|---------|
| Основной режим | Подтверждать **всю накладную целиком** одной кнопкой |
| Редактирование до подтверждения | Можно исправить позиции, уточнить модель/цвет, убрать ошибочный скан |
| На будущее | Возможность подтверждать **по одной позиции** — заложить в архитектуру, реализовать позже при необходимости |
| Неподтверждённые накладные | Висят в очереди **без таймаута** до ручного решения: подтвердить или удалить |

## 5. Связь между телефоном и ПК

### 5.1. Основной канал — облако

- Телефон и ПК подключаются к **одной облачной базе**.
- Не важно, через что идёт связь: Wi‑Fi, мобильная сеть (4G/5G), кабель на ПК.
- Изменения синхронизируются **в реальном времени** (запрос с телефона сразу виден на ПК).

### 5.2. Запасной канал — локальная сеть

Если облако недоступно, но телефон и ПК **в одной локальной сети** (или телефон подключён к hotspot ПК):

- телефон переключается на **прямое соединение с ПК**;
- сканирование и предзапросы работают как обычно;
- подтверждение — на ПК;
- при восстановлении облака данные **синхронизируются** с сервером.

### 5.3. Что НЕ используется

- **Bluetooth** — не требуется и не планируется.

### 5.4. Офлайн на телефоне

Если на складе временно нет сети:

- сканы сохраняются в **локальную очередь** на телефоне;
- при появлении связи (облако или локальная сеть) очередь отправляется на синхронизацию.

## 6. Данные и сущности

### 6.1. Справочники

| Сущность | Описание |
|----------|----------|
| **Изделие** | Комод, тумба, шкаф, холодильник и т.д. — модель + цвет |
| **Тип пакета** | Шаблон пакета (может подходить к нескольким изделиям) |
| **Рецепт комплекта (BOM)** | Для изделия: список типов пакетов, количество, цвет |
| **Единица товара** | Конкретная коробка с уникальной этикеткой |
| **Накладная** | Документ поступления или отгрузки (подтверждается целиком на ПК) |
| **Запрос операции** | Позиция в накладной, созданная сканом с телефона (до подтверждения на ПК) |
| **Заказ / клиент** | Обязательные поля накладной отгрузки |

### 6.2. Связи

```
Изделие ←──(рецепт)──→ Тип пакета     [многие-ко-многим]
Тип пакета ←──(экземпляр)──→ Единица товара (уникальная этикетка)
Накладная ←──→ Позиции ←──→ Единицы товара
Накладная отгрузки ──→ номер заказа + клиент
Запрос операции ←──→ Единица товара + тип (приём / отгрузка)
```

**Рецепты комплектов (BOM):** вводятся **вручную** на ПК. Должна быть возможность **добавлять и редактировать** в любой момент (изменить состав комода, добавить новую модель). Импорт из файла — на будущее (этап 4).

### 6.3. Статусы единицы товара

| Статус | Значение |
|--------|----------|
| `ожидает приёмки` | Отсканировано на телефоне, ждёт подтверждения на ПК |
| `на складе` | Принято, доступно |
| `привязано к изделию` | Пакет мебели назначен на конкретное изделие по накладной |
| `ожидает отгрузки` | Запрос на отгрузку создан, ждёт подтверждения на ПК |
| `отгружено` | Списано со склада |

## 7. Интерфейсы

### 7.1. Телефон (PWA / мобильный браузер)

| Экран | Функции |
|-------|---------|
| Главная | Режим: Приёмка / Отгрузка / Просмотр / Статус связи |
| Сканер | Камера: штрихкод, QR; поле ручного ввода кода |
| Результат скана | Модель, цвет, остаток; кнопка «Отправить запрос» |
| Очередь | Мои запросы: ожидает / подтверждено / отклонено |
| Связь | Облако: онлайн / офлайн; локальный ПК: доступен / нет |

### 7.2. ПК (браузер)

| Экран | Функции |
|-------|---------|
| Дашборд | Остатки, запросы в ожидании, неполные комплекты |
| Очередь / накладные | Накладные в ожидании → правка позиций → **Подтвердить всю накладную** |
| Накладные отгрузки | Номер заказа, клиент, список отсканированных позиций |
| Справочник изделий | Модели, цвета, рецепты комплектов (ручной ввод и редактирование) |
| Правила этикеток | Формат кода (`KOM-204-DUB`), расшифровка частей — с возможностью правки |
| Склад | Поиск и фильтры по коду, модели, цвету, статусу (рассчитано на **сотни** моделей) |
| Отчёты | Остатки, история операций, «чего не хватает к заказу» |
| Настройки | Склад, пользователи, синхронизация, локальный сервер |

## 8. Технические требования

### 8.1. Платформы

| Этап | Платформа |
|------|-----------|
| MVP | Веб/PWA: **Android**-телефон + ПК (браузер) |
| Позже | Нативное Android-приложение (при необходимости); iPhone — не в приоритете |

### 8.2. Стек (предварительно)

| Компонент | Технология |
|-----------|------------|
| Фронтенд | HTML, CSS, JavaScript (PWA) |
| Сканирование | Camera API + библиотека штрихкодов/QR |
| Облако | Firebase (Firestore) — отдельный конфиг Sklad, без связи с другими приложениями |
| Локальный режим | Локальный HTTP-сервер на ПК + автообнаружение в сети |
| Хостинг | Статика + облачная БД |

### 8.3. Требования к сети

- стабильный интернет на складе — желательно, но не обязателен (есть офлайн-очередь и локальный режим);
- для локального режима: телефон и ПК в одной Wi‑Fi сети или hotspot.

## 9. Этапы разработки

- [ ] Справочник изделий (ручной ввод)

- [ ] Сканирование в рамках накладной (приём / отгрузка) с телефона
- [ ] Подтверждение **всей накладной** на ПК (приёмка и отгрузка)
- [ ] Редактирование позиций накладной до подтверждения
- [ ] Отгрузка: поля **номер заказа** и **клиент**
- [ ] Остатки на складе
- [ ] Синхронизация через облако в реальном времени
- [ ] Базовые правила разбора этикеток (`KOM-204-DUB`) с редактированием на ПК

### Этап 2 — Мебель и накладные

- [ ] Рецепты комплектов — ручной ввод и редактирование на ПК
- [ ] Приёмка пакетов с последующей привязкой к изделию по накладной
- [ ] Отображение «чего не хватает» для комплекта

### Этап 3 — Надёжность и локальный режим

- [ ] Офлайн-очередь сканов на телефоне

### Этап 4 — Расширение

- [ ] Подтверждение по одной позиции (если понадобится)

- [ ] Импорт справочников из Excel/CSV
- [ ] Несколько складов
- [ ] Роли и несколько пользователей

| 1 | Подтверждение на ПК | **Всю накладную целиком**; подтверждение по одной позиции — **заложить на будущее** |
| 2 | Отгрузка: заказ и клиент | **Да** — оба поля обязательны |
| 3 | Телефон кладовщика | **Android** |
| 4 | Формат этикетки | Пример `KOM-204-DUB`; правила разбора — **с возможностью редактирования** на ПК |
| 5 | Рецепты комплектов («из чего комод») | **Вручную** на ПК; **можно дополнять и править** в любой момент |

| 7 | Объём моделей (мебель + техника) | **До сотен** — нужны поиск, фильтры и удобное редактирование справочника |
| 8 | Неподтверждённые накладные | **Хранить до решения оператора** — вручную **подтвердить** или **удалить**; автоматического срока истечения нет |

## 11. Критерии приёмки MVP (этап 1)

1. Кладовщик на **Android** сканирует этикетки → позиции попадают в накладную на ПК в течение нескольких секунд.
2. Оператор на ПК подтверждает **всю накладную приёмки** → остатки обновляются; на телефоне статус «Подтверждено».
3. При отгрузке на ПК заполняются **номер заказа** и **клиент** → подтверждение всей накладной → остатки уменьшаются.
4. До подтверждения можно **отредактировать** позиции накладной.
5. Отклонённая накладная не меняет остатки.
6. Код `KOM-204-DUB` разбирается на модель и цвет; правила редактируются на ПК.
7. Система работает в браузере на Android и ПК (PWA, без магазина приложений).

## 12. Ограничения и допущения

- Bluetooth не используется.
- Система не является пиратским каталогом контента; учитываются только товары заказчика с физическими этикетками.
- Юридическая ответственность за корректность справочников (рецепты комплектов) лежит на заказчике.
- Первая версия — веб/PWA; нативное приложение — по результатам эксплуатации MVP.
